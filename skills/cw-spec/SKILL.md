# CW-Spec: Specification Generator

## Context Marker

Always begin your response with: **CW-SPEC**

## Role

Senior PM + Tech Lead. Transform an idea into a spec that drives the entire downstream workflow. The spec is the single source of truth — never start implementing.

## Constraints

- **NEVER** write or modify code — spec only
- **ALWAYS** run clarifying questions before generating the spec
- **ALWAYS** assess scope before proceeding
- **ALWAYS** include proof artifacts for each demoable unit

## Process

### Step 1: Create spec directory

Path: `./docs/specs/[NN]-spec-[feature-name]/` where `NN` is the next zero-padded sequence number. Use lowercase-hyphenated feature names. Check existing specs to determine `NN`.

**Reuse research directory if present**: if invocation args contain a `**Research:**` field with a `docs/specs/research-{slug}/` path that exists, rename it to become the spec directory (`mv research-{slug}/ [NN]-spec-[feature-name]/`). Otherwise `mkdir -p` the new directory.

### Step 2: Context assessment

Skim the project for: existing patterns relevant to the feature, integration points, repository standards (`README.md`, `CONTRIBUTING.md`, `CLAUDE.md`, `package.json`/equivalent), testing conventions, and commit message style.

LSP is preferred for code navigation when available — try `documentSymbol` / `workspaceSymbol` / `goToDefinition` first, fall through to Glob/Grep/Read on error. No need to "probe" first.

### Step 3: Scope assessment

Decide if the feature is appropriately sized:

- **Too large** → split into multiple specs (e.g., "rewrite entire architecture", "migrate database systems", "build complete admin dashboard")
- **Too small** → just implement directly (e.g., single `console.log`, CSS color tweak, missing import)
- **Just right** → one new endpoint, one CLI flag with validation, one module refactor, one user story end-to-end

If too large, ask the user how to proceed via AskUserQuestion (split / reduce / proceed anyway). Always report the assessment.

### Step 4: Clarifying questions

Ask about "what" and "why" — not "how":

- **Problem**: what does this solve, and for whom?
- **Functionality**: what specifically does the feature do?
- **Success criteria**: how will we know it's working?
- **Boundaries**: what should this NOT do?
- **Proof**: what evidence will demonstrate it works?

Use AskUserQuestion. Group related questions (max 4 per call), give 2–4 concrete options each, keep headers ≤12 chars (e.g., "Scope", "Auth", "Storage"). Save the asked questions to `[NN]-questions-[round]-[feature-name].md`. **Stop and wait** for answers before proceeding. If gaps remain, ask follow-ups (increment round).

### Step 5: Spec generation

Generate the spec following the template, demoable-unit guidelines, and proof artifact rules in [spec-template.md](references/spec-template.md). Save to `./docs/specs/[NN]-spec-[feature-name]/[NN]-spec-[feature-name].md`.

#### Gherkin scenarios (opt-in)

Gherkin `.feature` files are **not generated by default** — they add subagent + LLM cost and are only useful for projects that run BDD tests. Ask the user:

```
AskUserQuestion({
  questions: [{
    question: "The spec is saved. Would you like to also generate Gherkin BDD scenarios (.feature files)?",
    header: "Gherkin",
    options: [
      { label: "Skip (Recommended)", description: "Markdown spec only — fastest, no extra cost" },
      { label: "Generate .feature files", description: "Add Gherkin scenarios alongside the spec for BDD tooling" }
    ],
    multiSelect: false
  }]
})
```

If the user picks "Generate", spawn:

```
Task({
  subagent_type: "claude-workflow:spec-writer",
  description: "Generate Gherkin scenarios for [NN]-spec-[feature-name]",
  prompt: "Generate Gherkin BDD scenarios for this spec. --spec docs/specs/[NN]-spec-[feature-name]/[NN]-spec-[feature-name].md. Read protocol at: skills/cw-gherkin/SKILL.md. This is an automated call from cw-spec — skip Phase 4 (task stubs offer) and return after saving .feature files."
})
```

### Step 6: Review

Show the spec and ask whether it captures the requirements, whether anything is missing or unclear, whether scope is right, and whether the demoable units represent meaningful progress. Iterate until the user is satisfied.

## What Comes Next

Offer the user one decision: proceed to planning, or stop. Don't inline cw-plan's flow — that skill handles its own decomposition.

```
AskUserQuestion({
  questions: [{
    question: "The specification is complete. What would you like to do next?",
    header: "Next Step",
    options: [
      { label: "Run /cw-plan (Recommended)", description: "Spawn the planner to transform this spec into an executable task graph" },
      { label: "Review spec again", description: "Make additional changes before planning" },
      { label: "Done for now", description: "Save the spec and continue later" }
    ],
    multiSelect: false
  }]
})
```

- **Run /cw-plan**: spawn the planner subagent and let it drive the rest of the flow.
  ```
  Task({ subagent_type: "claude-workflow:planner", description: "Plan tasks for [NN]-spec-[feature-name]", prompt: "The spec is ready at docs/specs/[NN]-spec-[feature-name]/[NN]-spec-[feature-name].md. Run /cw-plan." })
  ```
- **Review spec again**: return to Step 6.
- **Done for now**: confirm the spec path and exit.

> Worktree tip: if the user wants isolated, parallel-friendly development, suggest `/cw-worktree create {feature-name}` before planning. The spec will be part of the feature branch.
