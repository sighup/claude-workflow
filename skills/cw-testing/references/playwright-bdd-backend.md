# playwright-bdd Backend

Reference for the `playwright-bdd` automation backend in cw-testing.

## Overview

The `playwright-bdd` backend compiles `.feature` files into TypeScript test specs before execution. Tests run headlessly via Bash — no AI agent involvement during execution. This makes tests reproducible and CI-compatible.

### Execution Flow

```
.feature files
      │
      ▼
playwright.config.ts  (defineBddConfig pointing to features + step files)
      │
      ▼
npx bddgen            (generates .features-gen/*.spec.ts)
      │
      └─ Exit non-zero + print TypeScript scaffolds if any step lacks a definition
      │
      ▼
npx playwright test   (runs the generated spec files, produces JSON + HTML reports)
```

**Critical constraint**: `bddgen` exits non-zero if any step in any `.feature` file lacks a matching TypeScript implementation. There is no "run with pending steps" mode. All steps must be implemented before execution can proceed.

## Setup Procedure

When setting up the `playwright-bdd` backend for a spec:

1. **Check prerequisites** — verify `@playwright/test` is installed:
   ```bash
   npx playwright --version 2>/dev/null
   ```
   If missing, inform the user: "Run `npm install -g @playwright/test` (global) or `npm install @playwright/test` in your project root, then retry."

2. **Generate `playwright.config.ts`** — write to `[artifacts_dir]/playwright.config.ts` using the template below. If the file already exists, ask the user to confirm overwrite before writing.

3. **Generate step definitions** — spawn an implementer sub-agent:
   ```
   Task({
     subagent_type: "claude-workflow:implementer",
     prompt: "Read all .feature files in [gherkin_dir]. For each unique Given/When/Then step across all feature files, write a TypeScript step definition in [artifacts_dir]/steps/[feature-name].steps.ts using playwright-bdd's createBdd() pattern. Use semantic Playwright locators (getByRole, getByLabel, getByText). Feature files are at: [list of .feature file paths]. Reference: skills/cw-testing/references/playwright-bdd-backend.md"
   })
   ```

4. **Update `.gitignore`** — append the entries from the `.gitignore` section below if not already present.

5. **Verify with bddgen** — run:
   ```bash
   npx bddgen --config [artifacts_dir]/playwright.config.ts
   ```
   - Exit 0 → setup complete; proceed
   - Exit non-zero → show the output (which includes TypeScript scaffolds for missing steps); prompt user to review the scaffolds and retry, or spawn an implementer sub-agent to fill in the missing step bodies before retrying

## playwright.config.ts Template

```typescript
import path from 'path';
import { defineConfig, devices } from '@playwright/test';
import { defineBddConfig, cucumberReporter } from 'playwright-bdd';

const testDir = defineBddConfig({
  features: '../*.feature',
  steps: 'steps/*.ts',
  importTestFrom: 'fixtures.ts',
  outputDir: '.features-gen',
});

export default defineConfig({
  testDir,
  reporter: [
    ['json', { outputFile: path.join(__dirname, 'results.json') }],
    ['html', { outputFile: path.join(__dirname, 'report.html'), open: 'never' }],
  ],
  use: {
    screenshot: 'on',
    trace: 'on',
  },
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
  ],
});
```

The `features: '../*.feature'` path is relative to `playwright.config.ts` inside `testing/`, pointing back up to the spec directory where `.feature` files live.

## Custom Fixtures

Custom fixtures extend the base `playwright-bdd` test object with additional capabilities available to all step definitions. The `fixtures.ts` file lives at the `[artifacts_dir]` root alongside `playwright.config.ts`.

### fixtures.ts Template

```typescript
import { test as base } from 'playwright-bdd';

export const test = base.extend<{ consoleErrors: string[] }>({
  consoleErrors: async ({ page }, use) => {
    const errors: string[] = [];
    page.on('console', (msg) => {
      if (msg.type() === 'error') {
        errors.push(msg.text());
      }
    });
    await use(errors);
  },
});
```

The `importTestFrom: 'fixtures.ts'` line in `playwright.config.ts` tells `bddgen` to use this custom test object in all generated spec files, making `consoleErrors` available as a fixture parameter in any step definition.

When using custom fixtures, **all step files** must import from `fixtures.ts` and pass the custom `test` to `createBdd()`:

```typescript
import { createBdd } from 'playwright-bdd';
import { test } from '../fixtures';

const { Given, When, Then } = createBdd(test);
```

## Step Definition Patterns

Use `createBdd()` with the custom `test` fixture to create type-safe step definitions:

```typescript
import { expect } from '@playwright/test';
import { createBdd } from 'playwright-bdd';
import { test } from '../fixtures';

const { Given, When, Then } = createBdd(test);

Given('I am on the login page', async ({ page }) => {
  await page.goto('/login');
});

When('I enter {string} as username and {string} as password', async ({ page }, username, password) => {
  await page.getByLabel('Username').fill(username);
  await page.getByLabel('Password').fill(password);
});

When('I click the Login button', async ({ page }) => {
  await page.getByRole('button', { name: 'Login' }).click();
});

Then('I should see the dashboard', async ({ page }) => {
  await expect(page.getByRole('heading', { name: 'Dashboard' })).toBeVisible();
});
```

### Locator Preference Order

Use semantic locators in this order:
1. `getByRole` — for buttons, links, headings, inputs by ARIA role
2. `getByLabel` — for form fields by their label text
3. `getByText` — for visible text content
4. `getByPlaceholder` — for inputs by placeholder
5. `getByTestId` — for elements with `data-testid` attributes

Avoid CSS selectors and XPaths unless no semantic locator applies.

## Console Error Monitoring

A cross-cutting check that verifies pages render without JavaScript console errors. Uses the `consoleErrors` fixture defined in `fixtures.ts`.

### Feature File

```gherkin
# Pattern: Web/UI
# Recommended test type: E2E

Feature: Console Error Monitoring

  Scenario: Homepage renders without console errors
    Given I navigate to "/"
    Then there should be no console errors

  Scenario: Login page renders without console errors
    Given I navigate to "/login"
    Then there should be no console errors
```

Save as `console-error-monitoring.feature` in the spec directory alongside other `.feature` files.

### Step Definitions

Create `[artifacts_dir]/steps/console-errors.steps.ts`:

```typescript
import { expect } from '@playwright/test';
import { createBdd } from 'playwright-bdd';
import { test } from '../fixtures';

const { Given, Then } = createBdd(test);

Given('I navigate to {string}', async ({ page }, path: string) => {
  await page.goto(path);
});

Then('there should be no console errors', async ({ consoleErrors }) => {
  expect(
    consoleErrors,
    `Expected no console errors but found:\n${consoleErrors.join('\n')}`,
  ).toHaveLength(0);
});
```

The `consoleErrors` fixture automatically collects all `console.error()` messages from the moment the page loads. The `Then` step asserts the array is empty and prints the captured errors on failure.

### Combining with Other Scenarios

Console error checking can be added as a `Then` clause to any existing scenario:

```gherkin
  Scenario: Valid credentials redirect to dashboard
    Given the login page is displayed at /login
    When the user enters "user@example.com" and a valid password and clicks Login
    Then the page redirects to /dashboard
    And there should be no console errors
```

Because `consoleErrors` is a fixture, it collects errors across all page navigations and interactions within the scenario. No extra `Given` setup is needed — the listener attaches automatically.

## CLI Commands

### Generate spec files
```bash
npx bddgen --config docs/specs/<spec-name>/testing/playwright.config.ts
```

### Run a single scenario (cw-testing loop)
```bash
npx playwright test \
  --config docs/specs/<spec-name>/testing/playwright.config.ts \
  --grep "Exact Scenario Title" \
  --reporter=json
```

> Escape regex-special characters in the title: `( ) . [ ] * + ?` → prefix each with `\`

### Run the full suite (manual / debugging)
```bash
npx playwright test --config docs/specs/<spec-name>/testing/playwright.config.ts
```

### Run in headed mode (debugging)
```bash
npx playwright test \
  --config docs/specs/<spec-name>/testing/playwright.config.ts \
  --headed
```

## Result JSON Structure

Playwright's JSON reporter writes `[artifacts_dir]/results.json`. Key fields for parsing pass/fail:

```json
{
  "suites": [
    {
      "title": "Feature Title",
      "specs": [
        {
          "title": "Scenario title",
          "ok": true,
          "tests": [
            {
              "status": "expected",
              "results": [
                {
                  "status": "passed",
                  "error": null,
                  "attachments": [
                    {
                      "name": "screenshot",
                      "contentType": "image/png",
                      "path": "/abs/path/to/test-results/scenario-chromium/screenshot.png"
                    },
                    {
                      "name": "trace",
                      "contentType": "application/zip",
                      "path": "/abs/path/to/test-results/scenario-chromium/trace.zip"
                    }
                  ]
                }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

To determine pass/fail per scenario:
- `spec.ok == true` → scenario passed
- `spec.ok == false` → scenario failed; extract `tests[0].results[0].error.message` for `failure_reason`

To extract screenshots: filter `tests[0].results[0].attachments` where `contentType == "image/png"` and collect `path` values. Store in `artifacts: { screenshots: [...] }` on the step task.

## Common Errors

### Missing step definitions
```
Error: Step "I click the login button" is not defined.
```
**Fix**: `bddgen` prints TypeScript scaffolds for all missing steps. Copy the scaffold into the appropriate `steps/*.ts` file and implement the body.

### @playwright/test not found
```
Cannot find module '@playwright/test'
```
**Fix**: Install globally with `npm install -g @playwright/test` or locally with `npm install @playwright/test` in the project root.

### bddgen not found
```
command not found: bddgen
```
**Fix**: Install globally with `npm install -g playwright-bdd` or ensure `node_modules/.bin/bddgen` exists locally.

### playwright.config.ts path mismatch
If `bddgen` cannot find feature files, verify that the `features` glob in `defineBddConfig` is correct relative to the config file location. For the standard layout (`testing/playwright.config.ts` pointing to `../*.feature`), the path is `../\*.feature`.

## .gitignore

When setting up `playwright-bdd`, append these entries to the project's `.gitignore` if they are not already present:

```gitignore
# Playwright / playwright-bdd
.features-gen/
test-results/
blob-report/
playwright/.auth/
```

| Pattern | Contents |
|---------|----------|
| `.features-gen/` | Compiled specs generated by `bddgen` on every run |
| `test-results/` | Screenshots, traces, and videos from test runs |
| `blob-report/` | Intermediate results for sharded test runs |
| `playwright/.auth/` | Browser auth state — contains sensitive cookies/tokens |
