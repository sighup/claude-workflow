Nine stages, one task board

Each stage is its own slash command / skill. Solid nodes run every time; dashed nodes are optional add-ons you can slot in without changing the rest of the flow.

cw-researchsurvey the codebase

Explores the codebase across five dimensions and writes a structured report that seeds the spec with real context instead of assumptions.

cw-specwrite the specification

Produces demoable units, functional requirements, and proof-artifact definitions — the contract every later stage checks its work against.

cw-gherkinderive BDD scenarios

Called automatically from cw-spec. Emits one .feature file per demoable unit and, optionally, matching cw-testing task stubs.

cw-planbuild the task graph

Transforms the spec into dependency-aware tasks on the native Claude Code task board, each carrying self-contained execution metadata.

cw-dispatch / cw-executerun the work

Independent tasks fan out to parallel subagent workers (no setup) or a persistent agent team (opt-in). cw-execute drives a single task through an 11-step orient-to-commit protocol.

cw-reviewcatch what execution missed

Reviews for bugs, security issues, and spec compliance — file-partitioned or, as a team, concern-partitioned across security/correctness/compliance lenses — and opens fix tasks.

cw-testingprove it end-to-end

Generates tests from the spec, runs them in isolated sub-agents, and auto-fixes application bugs the tests surface.

cw-validategate on evidence

Runs seven validation gates against the coverage matrix — the last checkpoint before a PR, and the one nothing skips.

03 — the unit of work

Every task is self-contained

Tasks live on Claude Code's native task board, not a bespoke store. Each one carries everything an agent needs to execute it without re-reading the whole spec.

task metadata · T01 
{
  "task_id": "T01",
  "spec_path": "docs/specs/01-spec-auth/01-spec-auth.md",
  "scope": {
    "files_to_create": ["src/auth/login.ts"],
    "files_to_modify": ["src/routes/index.ts"],
    "patterns_to_follow": ["src/routes/health.ts"]
  },
  "requirements": [
    { "id": "R01.1", "text": "POST /auth/login accepts credentials", "testable": true }
  ],
  "proof_artifacts": [
    { "type": "test", "command": "npm test -- src/auth/login.test.ts", "expected": "All pass" }
  ],
  "commit": { "template": "feat(auth): add login endpoint" },
  "verification": { "pre": ["npm run lint"], "post": ["npm test"] },
  "role": "implementer",
  "complexity": "standard"
}

Scope bounds what a worker may touch. Requirements and proof artifacts are the contract cw-validate checks against. Commit template and verification commands mean a task can close itself — orient, implement, prove, commit — without a human relaying instructions between stages.

05 — parallel development

Worktrees give every feature its own task list

/cw-worktree creates an isolated git worktree and feature branch, and stamps its .claude/settings.local.json with a dedicated CLAUDE_CODE_TASK_LIST_ID — so tasks for that feature persist under ~/.claude/tasks/{worktree-name}/ and resume cleanly across sessions, without colliding with any other feature's board.

Two dispatch models sit on top of that isolation: cw-dispatch spins up subagent workers with zero setup and is the default; cw-dispatch-team runs a persistent team with a coordinating lead, gated behind CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 for when tasks need cross-worker coordination rather than independent execution.
